# Intent: Lainclaw 飞书私聊 MVP 接入

## 背景
当前 Lainclaw 主要是 CLI/ask 流程驱动，尚未形成外部渠道入口。为了在生产中让用户直接在飞书私聊与机器人交互，需要先补齐最小闭环能力，并以可扩展方式接入现有 ask pipeline。

## 目标
1. 新增飞书私聊（DM）文本消息入口与回包能力。
2. 飞书事件仅支持文本，按最小可靠性标准完成 webhook 接入。
3. 私聊用户发送文本后可走已有的 `ask` 能力并返回 Lainclaw 回复。
4. 群聊不参与处理（先明确拒绝/忽略），避免范围蔓延。

## 非目标（Non-Goals）
1. 群聊（Group）消息接入与处理。
2. 飞书卡片、富文本、附件、图片、文件等高级消息能力。
3. 命令配对（pairing）、复杂权限治理、速率限制策略。
4. 全量插件化重构；以最小侵入接入为主。

## 验收标准（Acceptance Criteria）
1. 使用飞书私聊发送任意文本，能够触发 Lainclaw 并返回文本消息。
2. Webhook URL verification/challenge 请求返回符合飞书要求。
3. 飞书群聊事件不会导致服务误处理，需记录可观测的拒绝日志。
4. 私聊消息失败（如飞书 API 回包失败）不应导致主进程崩溃。
5. 私聊会话上下文在一次会话中可持续；新会话可正常创建。
